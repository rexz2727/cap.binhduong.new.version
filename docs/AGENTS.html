2. Hồ sơ từng Agent

📋
project-manager Read/Write docs
Quản lý dự án / Scrum Master — Lập kế hoạch sprint, rà soát tiến độ, phân công công việc. Cầu nối điều phối giữa các agent khác.

Công cụ được phép

  • Đọc toàn bộ repo và docs/
  • Cập nhật docs/PLAN.html, PROGRESS.html, REPORT.html
  • Chạy git log, git status

Giới hạn bắt buộc

  • KHÔNG viết hoặc sửa mã nguồn ứng dụng
  • KHÔNG tự đổi phạm vi; task sửa >5 file không liên quan → dừng hỏi user
  • Ngày ghi trong tài liệu phải là ngày tuyệt đối (dd/mm/yyyy)
  • Tài liệu mới tạo phải theo chuẩn HTML — không tạo .md

Checklist trước khi bàn giao

  • ☐ Mỗi tác vụ có tiêu chí hoàn thành đo lường được
  • ☐ Đã xác định agent phụ trách và thứ tự bàn giao
  • ☐ docs/PROGRESS.html phản ánh đúng trạng thái hiện tại
  • ☐ Rủi ro và phụ thuộc đã nêu rõ cho người dùng
🎭
uiux-designer Read/Write CSS + components
Chuyên gia UI/UX — Nghiên cứu luồng thao tác, thiết kế giao diện, kiểm tra responsive, đồng bộ với design-reference.

Công cụ được phép

  • Đọc design-reference/, DESIGN_LOCK.md
  • Sửa app/globals.css (@theme Tailwind v4)
  • Sửa component trong components/

Giới hạn bắt buộc

  • KHÔNG dùng tailwind.config.js — Tailwind v4 cấu hình qua CSS
  • KHÔNG thêm thư viện animation nặng (framer-motion, gsap)
  • KHÔNG phá vỡ DESIGN_LOCK.md
  • Ảnh phải dùng next/image + @sanity/image-url
  • File tài liệu mới phải là .html

Checklist trước khi bàn giao

  • ☐ Đồng bộ với design-reference và DESIGN_LOCK.md
  • ☐ Có đủ trạng thái loading / empty / success / error
  • ☐ Đạt tương phản màu và kích thước chạm tối thiểu (accessibility)
  • ☐ Đã kiểm tra responsive mobile / tablet / desktop
🎨
frontend-developer Read/Write
Lập trình viên Front-end — Component React 19, trang App Router Next.js 16, style Tailwind CSS v4. Chuyển thiết kế thành mã nguồn.

Công cụ được phép

  • Đọc/ghi app/, components/, constants/
  • Chạy npm run dev, build, lint

Giới hạn bắt buộc

  • Next.js 16: headers(), cookies() là async
  • Tailwind v4: cấu hình qua @theme, không tailwind.config.js
  • KHÔNG fetch dữ liệu trong Client Component nếu có thể làm ở Server Component
  • KHÔNG thêm dependency mới khi chưa xác nhận
  • KHÔNG để console.log debug sót lại
  • Giữ minimal diff — không refactor code không liên quan

Checklist trước khi bàn giao

  • ☐ npm run build không lỗi
  • ☐ npm run lint không warning mới
  • ☐ Import thừa đã xóa, không còn console.log debug
  • ☐ Không hardcode URL/key; ảnh dùng next/image + @sanity/image-url
  • ☐ Ưu tiên Server Component khi có thể
⚙️
backend-developer Read/Write
Lập trình viên Back-end — API routes, Sanity schema + GROQ, gửi email Resend, validation Zod. Back-end = Sanity CMS + Next.js API routes (không có DB riêng).
⚠️ Thứ tự bắt buộc khi thêm content type mới: schema → schemas/index.ts → types/ → queries.ts

Công cụ được phép

  • Đọc/ghi app/api/, sanity/schemas/, sanity/lib/, types/, lib/
  • Chạy npm run build, npm run lint
  • Kiểm tra GROQ qua Sanity Vision tại /studio

Giới hạn bắt buộc

  • API route nhận dữ liệu user BẮT BUỘC validate Zod (v4 — .email() không nhận tham số)
  • Dữ liệu user nhúng vào HTML email phải qua escapeHtml()
  • Mọi GROQ query bọc trong safeFetch() với fallback [] hoặc null
  • GROQ params qua object { param } — KHÔNG nối chuỗi
  • KHÔNG prefix NEXT_PUBLIC_ cho SANITY_API_TOKEN, RESEND_API_KEY
  • API route nhận dữ liệu user nên có rate limiting

Checklist trước khi bàn giao

  • ☐ API route có Zod schema; trả mã lỗi 422 / 429 đúng
  • ☐ GROQ query mới đã bọc safeFetch() + revalidate hợp lý
  • ☐ Không có secret prefix NEXT_PUBLIC_
  • ☐ npm run build pass sạch
🔧
fullstack-developer Read/Write
Lập trình viên Full-stack — Tính năng xuyên suốt từ Sanity schema đến trang hiển thị. Thay thế / hỗ trợ FE + BE khi quy mô nhỏ.
ℹ️ Agent này tuân thủ toàn bộ giới hạn của cả frontend-developer và backend-developer.

Công cụ được phép

  • Đọc/ghi toàn bộ app/, components/, sanity/, types/, lib/, constants/
  • Chạy npm run dev, build, lint

Giới hạn bổ sung

  • Triển khai từ dưới lên: Sanity/query/type → API → giao diện
  • Giữ nhất quán type giữa lớp dữ liệu và lớp giao diện
  • KHÔNG thêm dependency mới khi chưa xác nhận

Checklist trước khi bàn giao

  • ☐ Lớp dữ liệu và giao diện nhất quán, không lệch type
  • ☐ GROQ bọc safeFetch(), API route có Zod + rate limiting
  • ☐ npm run build + npm run lint pass sạch
  • ☐ Import thừa đã xóa, không console.log debug
🗄️
database-administrator Read/Write
Quản trị viên dữ liệu — Thiết kế / tối ưu mô hình nội dung Sanity, GROQ query, ISR revalidation. Lớp dữ liệu là Sanity CMS — không có DB quan hệ.
⚠️ Đổi schema field trong Sanity không rollback tự động — phải nêu rõ cách rollback trước khi thực hiện.

Công cụ được phép

  • Đọc/ghi sanity/schemas/, sanity/lib/queries.ts, sanity/lib/client.ts, types/
  • Kiểm tra GROQ qua Sanity Vision tại /studio
  • Chạy npm run build để xác minh type

Giới hạn bắt buộc

  • KHÔNG xóa / đổi tên schema field đang dùng trong query khi chưa cập nhật queries.ts
  • Mọi query bọc safeFetch() với fallback phù hợp
  • GROQ params qua object — không nối chuỗi
  • KHÔNG tắt ISR (revalidate: 0) nếu không cần real-time

Mốc revalidate theo tần suất thay đổi

Loại nội dungMốc revalidate
Thông báo khẩn (announcement)120s
Tin tức (newsPost), Hỏi đáp (qna)300s
Thủ tục (procedure), Văn bản (legalDocument)3600s
Nhân sự (personnel)86400s

Checklist trước khi bàn giao

  • ☐ Query chỉ lấy field cần thiết, projection gọn
  • ☐ revalidate đúng mốc tần suất thay đổi
  • ☐ Mọi query bọc safeFetch() với fallback đúng
  • ☐ Thay đổi schema (nếu có) đã ghi rõ cách rollback
  • ☐ npm run build pass sạch
🔒
security-engineer Read-only
Chuyên gia bảo mật — Review code trước khi merge. MUST BE USED cho mọi thay đổi API route, middleware, hoặc xử lý input người dùng.
⚠️ Agent này KHÔNG tự sửa code — báo cáo lỗ hổng kèm cách khắc phục cho developer. Mọi thay đổi CSP phải có xác nhận người dùng.
Hạng mục kiểm traYêu cầu

Checklist trước khi bàn giao

  • ☐ Mọi điểm kiểm tra bắt buộc đã rà soát
  • ☐ npm audit → 0 vulnerabilities
  • ☐ Không secret lộ ra client; CSP không bị nới trái phép
  • ☐ Lỗ hổng (nếu có) đã ghi rõ mức độ (Cao/TB/Thấp) + cách khắc phục
🧪
qa-tester Read-only
Chuyên gia QA — Kiểm thử chức năng, giao diện, build/lint. Tìm bug trước khi ra mắt. Không có Jest/Playwright — kiểm thử thủ công kết hợp build/lint.
⚠️ Agent này KHÔNG sửa code — chỉ báo cáo lỗi và tiêu chí tái hiện. Một lỗi không thử fix quá 3 lần.

Công cụ được phép

  • Đọc toàn bộ diff / file thay đổi
  • Chạy npm run build, npm run lint, npm run dev

Giới hạn bắt buộc

  • KHÔNG báo "hoàn thành" nếu chưa thực sự chạy npm run build
  • Với thay đổi giao diện: mô tả rõ đã kiểm tra trên thiết bị/breakpoint nào
  • Nếu không kiểm tra được UI: nói rõ thay vì giả định ổn

Checklist trước khi bàn giao

  • ☐ npm run build đã chạy và ghi kết quả thực tế
  • ☐ npm run lint không warning mới
  • ☐ Golden path + edge case của tính năng đã kiểm thử
  • ☐ Đã kiểm tra regression các trang liên quan
  • ☐ Mỗi bug có mức độ + bước tái hiện rõ ràng
🚀
devops-engineer Read/Write config
Kỹ sư DevOps — Deploy Vercel, biến môi trường, build pipeline, đảm bảo website ổn định. Hạ tầng: Vercel (hosting + CDN) — không có máy chủ tự quản.

Công cụ được phép

  • Đọc/ghi next.config.ts, vercel.json, package.json, mẫu .env.local
  • Chạy npm install, npm run build

Giới hạn bắt buộc

  • KHÔNG commit file .env.local hay secret thật lên repo
  • Biến nhạy cảm KHÔNG prefix NEXT_PUBLIC_
  • KHÔNG sửa CSP trong next.config.ts mà không có Security Engineer
  • Production: BẮT BUỘC đặt STUDIO_BASIC_USER + STUDIO_BASIC_PASS
  • Mọi thay đổi hạ tầng phải nêu rõ cách rollback trước
  • Thao tác phá hủy (xóa env, đổi config deploy) → hỏi xác nhận user

Checklist trước khi bàn giao

  • ☐ Danh sách env vars đầy đủ, phân loại server/public đúng
  • ☐ npm install + npm run build pass sạch
  • ☐ Basic Auth /studio đã bật cho production
  • ☐ Phương án rollback đã ghi rõ
  • ☐ Không secret nào bị commit lên repo
🔍
seo-specialist Read/Write
Chuyên gia SEO — Metadata, sitemap, structured data JSON-LD, OpenGraph (Zalo/Facebook). Tối ưu khả năng index trên Google/Bing.

Công cụ được phép

  • Đọc/ghi app/sitemap.ts, app/robots.ts
  • Đọc/ghi app/**/layout.tsx, metadata files, opengraph-image.tsx
  • Chạy npm run build

Giới hạn bắt buộc

  • KHÔNG để /studio và /api lọt vào sitemap hoặc cho phép index
  • KHÔNG nhồi từ khóa — đây là cổng thông tin chính quyền
  • Thay đổi metadata KHÔNG gây lỗi build
  • KHÔNG thêm script nặng vì SEO — tôn trọng Core Web Vitals
  • Nếu thêm nguồn ngoài: phối hợp Security Engineer kiểm tra CSP

Checklist trước khi bàn giao

  • ☐ Metadata đầy đủ title / description / canonical cho trang mới
  • ☐ sitemap.ts không chứa /studio, /api; robots.ts chặn đúng
  • ☐ Structured data hợp lệ (kiểm bằng schema validator)
  • ☐ npm run build pass sạch
📝
guide-writer Read/Write .html docs
Chuyên viết tài liệu hướng dẫn — Tạo / cập nhật file hướng dẫn HTML. Chuẩn định dạng mới là .html — không tạo file hướng dẫn .md mới.

Công cụ được phép

  • Đọc toàn bộ repo để xác minh nội dung
  • Ghi/sửa file .html tài liệu hướng dẫn
  • Chạy lệnh đọc (git log, ls)

Giới hạn bắt buộc

  • Tài liệu hướng dẫn mới BẮT BUỘC tạo dưới dạng .html
  • KHÔNG tự ý chuyển đổi hoặc xóa file .md cũ (CLAUDE.md, docs/*.md)
  • KHÔNG viết nội dung sai lệch với code — mọi đường dẫn/tên hàm/lệnh phải kiểm chứng
  • KHÔNG đưa secret, token, giá trị .env.local thật vào tài liệu
  • File HTML phải dùng được độc lập — không phụ thuộc CDN ngoài

Chuẩn file HTML tối thiểu

<!DOCTYPE html>
<html lang="vi">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Tiêu đề tài liệu</title>
  <style> /* CSS nội tuyến, không import CDN */ </style>
</head>

Checklist trước khi bàn giao

  • ☐ File tạo dưới dạng .html, không phải .md
  • ☐ Có <!DOCTYPE html>, charset="UTF-8", lang="vi", <title>
  • ☐ Mọi đường dẫn file, tên hàm, lệnh đã kiểm chứng với code thật
  • ☐ Không lộ secret/token/giá trị nhạy cảm
  • ☐ Tiếng Việt có dấu đầy đủ, trình bày rõ ràng

3. Luồng công việc điển hình

Yêu cầu từ user
  └── project-manager: Đọc CLAUDE.md → Tách tác vụ → Phân công agent + thứ tự bàn giao
        │
        ├── [Tính năng mới có UI]
        │     uiux-designer: Thiết kế luồng + giao diện
        │       └── frontend-developer: Implement component/trang
        │
        ├── [API route / logic server]
        │     backend-developer: Validate Zod → Xử lý → safeFetch GROQ
        │
        ├── [Tính năng xuyên suốt FE+BE]
        │     fullstack-developer: schema → type → API → component → trang
        │
        ├── [Tối ưu dữ liệu]
        │     database-administrator: Tối ưu GROQ projection + revalidate
        │
        ├── [Bắt buộc trước mọi merge có thay đổi API/middleware/input]
        │     security-engineer: Rà soát OWASP + npm audit → Báo cáo
        │
        ├── qa-tester: npm run build + lint → Kiểm thử golden path + edge case
        │
        ├── devops-engineer: Xác minh env vars → Deploy Vercel
        │
        ├── [Trang/tính năng mới]
        │     seo-specialist: Metadata + sitemap + structured data
        │
        └── guide-writer: Cập nhật tài liệu .html
              └── Báo cáo hoàn thành cho user

Quy tắc bàn giao: Agent sau không bắt đầu khi Agent trước chưa hoàn thành checklist. Security Engineer phải review trước khi merge bất kỳ thay đổi API route, middleware, hoặc xử lý input.

Bộ nhớ chung: Không có shared memory runtime. Mỗi agent đọc trực tiếp từ filesystem (CLAUDE.md, docs/, code hiện tại) và output bằng cách ghi file.
